## 一、后端架构与设计模式（Q1-7）

### Q1. 后端整体架构
- 核心业务：自然语言 → AI生成网站 → 可视化修改 → 部署
- 9大模块：用户、应用、AI代码生成、对话与记忆、文件处理与构建、可视化编辑、AI工作流、部署、可观测性
- 交互流程：用户登录 → 创建应用 → 智能路由 → 对话生成 → 文件保存 → 部署 → 监控

### Q2. MyBatis Flex vs MyBatis Plus
- Flex更轻量（无第三方依赖）、性能更高（无拦截器/SQL解析）
- Flex原生支持多表关联查询
- 选型原因：丰富技术栈 + 性能优势

### Q3. 策略模式 — 代码解析
- 定义统一接口 `CodeParser<T>`，泛型支持不同返回类型
- 具体策略：HtmlCodeParser、MultiFileCodeParser
- 引入 `CodeParserExecutor` 执行器统一调度（执行器+策略模式组合）
- 扩展：新增策略类 + executor加case，符合开闭原则

### Q4. 模板方法模式 — 文件保存
- 抽象类 `CodeFileSaverTemplate<T>` 定义 final 的 saveCode 模板方法
- 固定流程：验证输入 → 创建目录 → 保存文件（可变步骤）
- 子类：HtmlCodeFileSaver（单文件）、MultiFileCodeFileSaver（三文件）
- 同样通过执行器调用，屏蔽差异

### Q5. Selenium + WebDriverManager
- WebDriver：程序与浏览器之间的桥梁/标准化接口
- WebDriverManager：自动检测浏览器版本 → 自动下载驱动 → 自动配置路径 → 缓存管理

### Q6. 集中式工具管理
- 抽象基类 `BaseTool` + Spring依赖注入 + `ToolManager`
- ToolManager通过 @Resource 注入 BaseTool 数组，@PostConstruct 建Map
- 提供 getTool(name) 和 getAllTools() 方法
- 新增工具：继承BaseTool + @Component，无需改管理代码

### Q7. 多模式生成的架构隔离
- 门面模式：`AiCodeGeneratorFacade` 统一入口
- 策略模式：不同模式对应不同系统提示词 + AI Service配置
- 工厂模式：`AiCodeGeneratorServiceFactory` 按类型构建实例（Vue用推理模型+工具，HTML用对话模型无工具）
- `StreamHandlerExecutor` 隔离流处理逻辑

## 二、AI 集成与 LangChain4j（Q8-17）

### Q8. 系统提示词设计技巧
- 精确角色扮演（资深前端专家）
- 严格约束（禁止外部依赖、必须用工具创建文件）
- 定义输出格式（Markdown代码块，便于正则提取）
- 处理迭代修改（必须返回完整文件）
- 迭代测试优化（先在大模型平台调试再应用）

### Q9. LangChain4j 集成方式
- AI Service模式：定义接口 → 注解驱动(@SystemMessage/@UserMessage/@MemoryId) → 工厂构建动态代理 → 业务层像调普通方法
- 其他特性：结构化输出、工具调用

### Q10. 结构化输出
- 方法返回值改为Java类，框架自动追加JSON指令
- 优化措施：增大max_tokens防截断、开启JSON模式(response-format)、@Description注解描述字段、提示词补充格式要求
- 兜底：框架自带重试机制

### Q11. Vue工程生成流程
- AI规划 → 多次调用writeFile工具 → 后端FileWriteTool写文件(vue_project_{appId}目录)
- 流式反馈：自定义JSON消息格式(AI_RESPONSE/TOOL_REQUEST/TOOL_EXECUTED)
- 自动构建：流结束异步触发 npm install + npm run build
- 预览/部署指向dist目录

### Q12. FileWriteTool 实现
- @Tool注解标记、@P注解描述参数
- @ToolMemoryId 自动注入appId实现目录隔离
- 路径：vue_project_{appId}/相对路径
- 安全：返回信息只含相对路径，不暴露绝对路径

### Q13. AI智能路由
- 专门的路由提示词 + 结构化输出(返回枚举CodeGenTypeEnum)
- 规则：简单需求→HTML，复杂需求→VUE_PROJECT
- 配置轻量级低成本模型（qwen-turbo）
- 结果存入数据库app表，后续流程据此执行

### Q14. 输入护轨 — 防注入
- 实现 `PromptSafetyInputGuardrail`
- 多维校验：长度限制 + 敏感词过滤 + 正则匹配注入模式
- 失败返回 fatal() 中断调用
- 通过 .inputGuardrails() 注册到AI Service

### Q15. 输出护轨 — 自动重试
- 实现 `RetryOutputGuardrail`
- 校验：空内容、长度过短、敏感信息泄露
- 不合格返回 reprompt("修正指令")，框架自动追加指令重新调用
- 合格返回 success()

### Q16. 多模型配置策略
- 核心思想：不同任务匹配不同模型，降本增效
- 复杂推理(Vue生成)：deepseek-reasoner，高max_tokens低temperature
- 简单分类(路由)：qwen-turbo，成本低十倍以上
- 实现：yml独立配置块 → @Configuration创建Bean → 工厂按场景注入

### Q17. 防止工具调用无限循环
- maxSequentialToolsInvocations(20) 设上限
- 提供 ExitTool 让AI主动退出
- 增大对话记忆上下文防止AI"断片"

## 三、对话与记忆（Q18-23）

### Q18. 游标分页 vs 传统分页
- 传统分页问题：数据插入/删除导致重复或遗漏 + 深页性能差(需跳过N行)
- 游标分页：WHERE createTime < 游标值，利用索引直接定位，性能稳定
- 适合实时性强、数据频繁变动的场景

### Q19. 对话历史表索引设计
- idx_appId：按应用筛选（删除关联记录）
- idx_createTime：按时间排序（管理员查询）
- idx_appId_createTime：核心复合索引，服务游标分页（先定位appId再按时间定位）

### Q20. Redis持久化对话记忆
- 性能优势（内存读写快）
- 避免JVM内存溢出
- 服务重启后保留状态
- 与业务数据隔离（chat_history表存完整记录，Redis存纯对话上下文）

### Q21. 对话记忆隔离
- 工厂动态创建AI Service实例（非单例）
- 每个实例绑定专属 MessageWindowChatMemory(id=appId)
- Redis Key: langchain4j:chat_memory:{appId}，物理隔离

### Q22. 从数据库加载历史到记忆
- 目的：Redis过期后恢复记忆，保证对话连续性
- 流程：创建AI Service时触发 → 查最近N条(跳过最新一条) → 清空旧记忆 → 反转后按时间正序逐条加载

### Q23. Caffeine缓存策略
- maximumSize(1000)：最大容量
- expireAfterWrite(30min)：写入后过期
- expireAfterAccess(10min)：访问后过期
- removalListener：移除日志监控

## 四、流式输出与SSE（Q36-38）

### Q36. 为什么选SSE
- SSE：服务器单向持续推送，基于单个HTTP长连接
- 解决AI生成耗时长用户等待问题
- 实现打字机效果，缩短感知等待时间
- LangChain4j的Flux对象天然支持

### Q37. SSE优化
- 数据丢失：每个数据块封装JSON对象 {"d":"..."} 传输
- 流结束：发送自定义 done 事件，区分正常结束和异常中断

### Q38. 工具调用流式输出
- 定义StreamMessage基类(type字段区分：ai_response/tool_request/tool_executed)
- 使用TokenStream替代Flux，监听工具调用中间状态
- JsonMessageStreamHandler处理器统一处理
- 优化：同一工具只推送一次"选择工具"提示，执行完毕后推送完整信息

## 五、可视化编辑（Q24-25, Q49-50）

### Q24. 可视化修改后端实现
- 前端捕获元素信息 → 拼接增强提示词 → 后端接收
- AI工具集：FileRead/FileModify/FileDelete/FileDirRead
- AI自主循环：读目录结构 → 读文件内容 → 调用FileModifyTool(路径+旧内容+新内容)

### Q25. 文件操作安全性
- 目录隔离：严格限制在应用专属目录，禁止 ../ 遍历
- 重要文件保护：不允许删除 package.json/vite.config.js/index.html 等
- 只允许删文件不允许删目录
- FileModifyTool基于旧内容替换（未找到则中止）
- try-catch + 日志审计

### Q49. 前端iframe跨窗口通信
- Vite代理保证同源 → 动态注入脚本到iframe
- 注入脚本：捕获阶段监听click → 阻止默认行为 → 收集元素信息 → postMessage发送给父窗口
- 父页面监听message事件获取选中元素

### Q50. 生成唯一CSS选择器
- generateSelector函数：从点击元素向上遍历DOM树
- 优先用#id作为起点（最稳定）
- 无ID则用标签名+类名+:nth-child(n)确保唯一
- 用 > 连接符拼接完整路径

## 六、AI工作流 LangGraph4j（Q26-28）

### Q26. 为什么引入LangGraph4j
- 之前的痛点：流程黑盒随机性大、任务耦合在一个提示词、扩展困难
- 解决：白盒标准化流程 → 节点单一职责 → 支持条件边和循环（质检失败重试）

### Q27. 工作流核心节点
- 共享状态：WorkflowContext
- 节点顺序：图片收集 → 提示词增强 → 智能路由 → 代码生成 → 质量检查 → 项目构建
- 每个节点从Context取输入、处理后更新回Context

### Q28. 图片收集并发调用
- AI规划服务返回结构化 ImageCollectionPlan
- CompletableFuture.supplyAsync() 为每个任务创建异步请求
- CompletableFuture.allOf().join() 等待全部完成
- 汇总结果更新到WorkflowContext
- 实测缩短70+秒

## 七、部署与运维（Q29-35, Q40-43, Q55）

### Q29. 应用封面图自动生成
- 部署成功后异步触发 → Selenium无头Chrome访问URL → 等待渲染 → 截图 → 压缩 → 上传COS → 更新数据库

### Q30. 截图异步执行
- 原因：耗时I/O操作，同步会阻塞用户
- 实现：Java 21虚拟线程（创建开销极低、I/O等待自动让出平台线程、API与传统线程一致）

### Q31. 代码打包下载
- Hutool ZipUtil.zip() 流式压缩到Response输出流（不落盘）
- 黑名单过滤：按名称(node_modules/dist/.git等) + 按扩展名(.log/.tmp/.cache)

### Q32. Spring Boot预览 + Nginx部署
- Spring Boot：实时预览，生成即可见，开发调试用
- Nginx：生产部署，高并发性能强，稳定可靠

### Q33. Nginx职责
- 静态资源托管（前端+用户生成应用）
- 反向代理（/api → Spring Boot）
- 适配SPA（try_files → index.html）
- 优化长连接（proxy_read_timeout 900s保SSE）

### Q34. AI调用性能瓶颈
- 根因：ChatModel单例 + 底层HTTP客户端同步阻塞
- 表现：并发退化为串行
- 解决：ChatModel Bean作用域改为prototype（多例）

### Q35. Redis使用场景
- 分布式Session（spring-session-data-redis）
- AI对话记忆持久化（RedisChatMemoryStore）
- 热点数据缓存（精选应用列表，Spring Cache）
- 分布式限流（Redisson RRateLimiter令牌桶）

### Q40. 可观测性
- 定义：通过外部输出推断内部状态
- 价值：快速定位问题、保障稳定性、优化AI调用成本、提升运维效率

### Q41. ARMS监控
- 探针技术：-javaagent挂载，字节码增强，零代码侵入
- 监控维度：接口性能/调用链追踪/异常分析 + 数据库SQL/Redis + JVM/系统资源

### Q42. ARMS vs Prometheus+Grafana
- ARMS：通用系统性能监控，自动无侵入
- Prometheus+Grafana：自定义业务指标（模型调用量/Token消耗/用户排行），需手动埋点
- 关系：互补，构建全方位可观测体系

### Q43. 自定义业务指标监控
- 采集：Micrometer(Counter/Timer) + ChatModelListener埋点 + /actuator/prometheus端点
- 存储：Prometheus定期拉取，TSDB存储
- 展示：Grafana仪表盘 + PromQL查询

### Q55. 完整部署流程
- 环境：1Panel + MySQL + Redis + Nginx + JDK21 + Node.js + Chrome
- 后端：application-prod.yml → Maven打包JAR → nohup启动
- 前端：.env.production → npm run build → dist上传
- Nginx：前端托管 + try_files + /api反向代理 + /dist别名映射

## 十、微服务架构（Q44, Q57, Q60-62）

### Q44. 微服务模块划分
- 通用模块：common(工具类)、model(实体/DTO/VO)、client(服务间接口契约)
- 业务服务：用户服务、应用服务、AI服务、截图服务（IO密集独立部署）

### Q57. 多人实时协作（设计题）
- WebSocket全双工通信
- 后端：会话池按appId分组 + 编辑锁机制 + 事件驱动广播
- 前端：WebSocket客户端 + 监听更新 + 根据锁状态切换UI
- 数据一致性：后端为唯一数据源

### Q58. Vibe Coding开发模式
- 优势：模式化工作效率极高、框架搭建快、聚焦高层思考
- 关键点：Prompt质量决定输出、需要强甄别调试能力、非确定性需版本控制+测试

### Q59. 最具挑战功能 — 可视化修改
- 难点：跨窗口通信 + AI交互 + 浏览器安全
- 方案选择：排除手动编辑器，采用"元素定位+AI修改"
- 同源处理：Vite代理
- 动态注入脚本（不污染源代码）
- HTML模式：提示词约束返回完整文件
- Vue模式：增量修改（AI用工具读+改）

### Q60. 单体→微服务 利弊
- 好处：独立部署、技术选型灵活、弹性扩展、故障隔离
- 问题：分布式复杂性、运维复杂、服务治理、数据一致性、开发成本增加

### Q61. Higress vs Spring Cloud Gateway
- Higress：C++/Envoy性能高、AI原生支持（缓存/令牌限流）、云原生设计
- SCG：JVM/WebFlux，通用但无AI特化能力

### Q62. 微服务拆分原则
- 业务领域驱动、数据一致性、团队组织(2-8人)、性能考虑(重计算独立)、扩展性需求
- 核心：灵活，耦合度高的不要硬拆
